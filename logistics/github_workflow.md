# GitHub Setup Guide

## Browser-Only Instructions for DDIT Students

**No terminal. No command line. No installation required.**

Everything in this guide is done entirely through a web browser at [github.com](https://github.com).

---

## What You Will Have When You Are Done

By the end of this guide you will have:

- A personal GitHub account
- Access to the course repository where all templates, frameworks, and case materials live
- Your own copy of the course materials that you can edit freely
- A shared team repository that all four team members can read and write
- The ability to create, edit, and upload files entirely from your browser

---

## Part 1. Create Your Personal GitHub Account

**Time required: 5 minutes**

1. Open your browser and go to [github.com](https://github.com).

2. Click the **Sign up** button in the top-right corner.

3. Enter your email address. Use your KIU student email if you have one, or any reliable personal email. You will need to access this email to verify your account.

4. Create a password. GitHub will tell you if it is strong enough.

5. Choose a username. This is public and permanent. Use something professional: your name or an abbreviated version of it. Examples: `mariamberekashvili`, `nani-danelia`, `saba-kepuladze`. Avoid numbers, random strings, or usernames that will embarrass you in a professional context.

6. Answer the setup question about whether you want product updates. Either answer is fine.

7. Click **Create account**.

8. GitHub will send a verification code to your email. Open your email, find the code, and enter it on the GitHub page.

9. On the next screen, GitHub asks some questions about your experience and intended use. You can answer or skip. When asked to choose a plan, select **Free**. Click **Continue for free**.

10. You now have a GitHub account. You will see your personal dashboard at `github.com/yourusername`.

**Share your GitHub username with the instructor and with your team members before leaving Weekend 1.**

## Part 2. Find and Access the Course Repository

**Time required: 2 minutes**

## Part 3. Copy the Course Materials to Your Own Account

**Why you need your own copy:** You cannot edit files directly in the instructor's repository. You need your own copy where you can save your work, experiment with templates, and build your deliverables.

There are two ways to do this in the browser. Use whichever matches your situation.

---

### Option A. Fork the Repository (Recommended)

Forking creates a complete copy of the course repository under your own GitHub account. Your copy stays linked to the original, so you can pull in updates the instructor makes during the semester.

**Time required: 2 minutes**

1. Make sure you are logged into GitHub.

2. Go to the course repository.

3. In the top-right corner of the repository page, click the **Fork** button. It has a small icon that looks like a branch split.

4. A dialog box appears. Under **Owner**, make sure your own username is selected. Under **Repository name**, you can keep the same name or change it. A clear name like `DDIT-2026-YourName` works well.

5. Leave **Copy the main branch only** checked.

6. Click **Create fork**.

7. GitHub creates your copy and redirects you to it. The address will be `github.com/yourusername/repositoryname`.

8. You will see a note at the top: *This branch is up to date with [instructor's repo].* That is correct.

**Your fork is now your personal working copy. Any changes you make here do not affect the course repository.**

---

### Option B. Download and Re-Upload (If Forking Is Not Available)

If the course repository is private and forking is disabled, use this method.

**Time required: 5 minutes**

1. Go to the course repository.

2. Click the green **Code** button near the top right of the file list.

3. In the dropdown, click **Download ZIP**.

4. Your browser downloads a zip file of the entire repository to your computer.

5. Unzip the file on your computer. You now have a folder with all the course materials.

6. Go to your GitHub account homepage (`github.com/yourusername`).

7. Click the **+** button in the top-right corner and select **New repository**.

8. Name it something like `DDIT-2026`. Leave all other settings as default. Select **Public** unless you have a reason to keep it private.

9. Click **Create repository**.

10. On the next screen, scroll down to the section **...or upload an existing file** and click the **uploading an existing file** link.

11. Drag the entire unzipped folder contents into the upload area, or click **choose your files** and select all files.

12. Scroll down, write a commit message like `Initial upload from course repo`, and click **Commit changes**.

13. Your copy is now on GitHub at `github.com/yourusername/DDIT-2026`.

## Part 4. Create Your Team Repository

**Time required: 10 minutes**

Your team needs one shared repository that all four members can edit. This is where your deliverables live: issue trees, hypothesis logs, competitive landscape, sprint plans, and presentation drafts.

**One person on the team creates the repository. The other three are added as collaborators. Decide who will do this before you start.**

---

### Step 1. Create the Team Repository

The designated team member does this:

1. Log into GitHub.

2. Click the **+** button in the top-right corner and select **New repository**.

3. For the repository name, use the format `TeamX-DDIT-2026` where X is your team letter. Examples: `TeamA-DDIT-2026`, `TeamB-DDIT-2026`, `TeamC-DDIT-2026`.

4. Add a short description. Example: `Team A — Retail Banking in Georgia. DDIT course, KIU FIM 2025-2026.`

5. Choose **Public**. This allows the instructor to review your work without needing special access. If your team has a strong reason to use Private, discuss with the instructor first.

6. Check the box that says **Add a README file**. This creates the first file automatically so the repository is not empty.

7. Leave all other settings as defaults.

8. Click **Create repository**.

9. Your team repository is now live at `github.com/yourusername/TeamX-DDIT-2026`.

---

### Step 2. Add Your Team Members as Collaborators

The person who created the repository does this:

1. Go to your team repository.

2. Click the **Settings** tab near the top of the repository page.

3. In the left sidebar, click **Collaborators**.

4. GitHub may ask you to confirm your password.

5. Click the **Add people** button.

6. In the search box, type each teammate's GitHub username. Select their name from the results.

7. Click **Add [username] to this repository**.

8. GitHub sends an invitation email to that person.

9. Repeat for all three remaining team members.

---

### Step 3. Each Team Member Accepts the Invitation

Each of the three invited team members does this:

1. Check the email address associated with your GitHub account. You will have an email from GitHub with the subject line: *[username] has invited you to collaborate on...*

2. Click the **Accept invitation** button in the email.

3. Alternatively: go to `github.com/notifications` and look for the invitation there.

4. Once accepted, you can go to `github.com/[repo-owner-username]/TeamX-DDIT-2026` and you will have full read and write access.

5. To make the repository easy to find later: on the repository page, click the **Star** button (top right) so it appears in your starred list, and click **Watch** so you get notifications when teammates make changes.

## Part 5. Set Up Your Team Folder Structure

**Time required: 10 minutes**

GitHub cannot create empty folders. You create a folder by creating a file inside it. This is a one-time setup task that the team Recorder should complete.

1. Go to your team repository.

2. Click **Add file** and then **Create new file**.

3. In the file name box at the top, type the following exactly, including the slash:
   ```
   working_files/README.md
   ```
   The slash tells GitHub to create a folder called `working_files` and put the file inside it.

4. In the file content area, type:
   ```
   # Working Files
   
   This folder contains drafts, research notes, and work in progress.
   ```

5. Scroll down to **Commit new file**. Write a commit message: `Create working_files folder`.

6. Click **Commit new file**.

7. Repeat this process to create three more folders. For each one: click **Add file**, **Create new file**, type the path shown below, add a one-line README, and commit.

   ```
   deliverables/README.md       (for final submissions)
   research/README.md           (for evidence, notes, source logs)
   presentations/README.md      (for slide decks and exports)
   ```

8. When done, your repository should show four folders plus the main README.

4. Click the **Raw** button at the top right of the file content. This shows the plain text of the file with no formatting applied.

6. Go to your team repository.

7. Navigate to the `working_files` folder.

9. In the name box, type the file name using the course naming convention:

10. Click into the content area and paste the template text with Ctrl+V or Cmd+V.

11. Replace the placeholder text in the template with your team's actual content.

12. Scroll down to **Commit new file**. Write a clear commit message:
    ```
    Add issue tree template for Team A
    ```

## Part 7. Edit Files in the Browser

You do not need any software to edit files on GitHub. Everything can be done in the browser.

1. Go to your team repository.

2. Navigate to the file you want to edit.

3. Click on the file name to open it.

4. Click the **pencil icon** (Edit this file) in the top right of the file content area.

5. The file opens in a text editor inside your browser. Make your changes.

6. When done, scroll down to the **Commit changes** section.

7. Write a clear commit message that describes what changed. Examples:
   - `Add hypotheses H1 and H2 with evidence plans`
   - `Update Porter Five Forces with banking sector data`
   - `Complete strategic problem statement draft`

8. Leave **Commit directly to the main branch** selected.

9. Click **Commit changes**.

10. Your changes are saved. Every edit is stored permanently in the commit history, so nothing is ever truly lost.

## Part 10. Working as a Team Without Conflicts

When multiple people edit the same file, GitHub tracks every change. Follow these four rules to avoid overwriting each other's work.

**Rule 1. Do not edit the same file at the same time.**

Before editing a file, check who last changed it. Click on the file, then click **History** to see recent commits and who made them.

**Rule 2. Write clear commit messages so teammates know what changed.**

Bad: `update`
Good: `Add hypotheses H4 to H6 to hypothesis log, update evidence plan for H1`

**Rule 3. Use your own named files for individual research notes.**

For personal research findings, create a file with your name in the title: for example `research/findings_mariam_20260322.md`. This prevents conflicts entirely because no two people are editing the same file.

**Rule 4. If two people edit the same file at the same time, GitHub will show a merge conflict.**

The file will display both versions separated by markers that look like `<<<<<<<`, `=======`, and `>>>>>>>`. The Recorder is responsible for resolving this. Open the file in the editor, read both versions, keep the correct combined content, delete all the marker lines, and commit with the message: `Resolve edit conflict in [filename]`.

## File Naming Quick Reference

Every file you submit must follow this format:

```
TeamX_Deliverable_v1_YYYYMMDD.md
```

| Part | What It Means | Example |
|---|---|---|
| `TeamX` | Your team letter | `TeamA`, `TeamB`, or `TeamC` |
| `Deliverable` | Clear descriptive name | `IssueTree`, `HypothesisLog`, `CompetitiveLandscape`, `DisruptionDiagnostic`, `ROIPrimer`, `TeamCharter`, `SprintPlan` |
| `v1` | Version number | Starts at `v1`, becomes `v2` when you submit a revised version |
| `YYYYMMDD` | Date in year-month-day format | `20260314` for 14 March 2026 |

Full examples:
```
TeamA_IssueTree_v1_20260314.md
TeamB_CompetitiveLandscape_v2_20260322.md
TeamC_HypothesisLog_v1_20260314.md
TeamA_ROIPrimer_v0_20260315.md
```

## Common Problems and Solutions

**I cannot find the invitation email.**
Go to `github.com` and log in. Click the bell icon in the top right. Your invitation should appear in your notifications. Alternatively go to `github.com/notifications` directly.

**I accidentally edited a file and want to undo the change.**
Click on the file, then click **History**. Find the commit just before your unwanted edit. Click on it, then click **Browse files** to see the earlier version. Open that version of the file, copy all the content, navigate back to the current file, click the pencil icon to edit, select all and replace with the older content, and commit with the message: `Revert [filename] to previous version`.

**My teammate and I edited the same file and there is now a conflict warning.**
GitHub marks both versions inside the file using `<<<<<<<`, `=======`, and `>>>>>>>` symbols. Click the pencil icon to edit the file. Read both versions carefully. Delete the markers and combine the correct content from each version. Commit with the message: `Resolve edit conflict in [filename]`. The Recorder role owns this.

**I cannot see the team repository after accepting the invitation.**
Go directly to the full URL: `github.com/[repo-owner-username]/TeamX-DDIT-2026`. If you still see an error, ask the repository owner to confirm the invitation was sent to the correct GitHub username, not just an email address.

**The file I uploaded is not showing in the repository.**
An upload is not saved until you click **Commit changes** at the bottom of the upload page. Go back, re-upload the file, and make sure to scroll down and click **Commit changes**.

**I cannot see the Edit button on a file.**
You must be logged in AND be a collaborator on the repository. If you can see the file but there is no pencil icon, check that you accepted the collaboration invitation correctly.

# GitHub Workflow Guide

## Why this matters

This repository is part course platform and part consulting workspace. Git gives your team version history, accountability, and a clean record of how your thinking evolved.

## Basic team workflow

1. Open your team directory.
2. Create or update files using the provided templates.
3. Commit changes with a clear message.
4. Push frequently.
5. Do not overwrite teammates' work blindly.

## File naming conventions

Use descriptive names.

Good:
- `issue_tree_v1.md`
- `hypothesis_log_week1.md`
- `industry_research_notes_2026-03-22.md`

Poor:
- `notes.md`
- `stuff.md`
- `finalfinal.md`

## Commit message examples

- `Add Team 2 issue tree v1`
- `Update competitive landscape with new entrant analysis`
- `Add March 22 hypothesis research notes`

## Team directory rule

Keep all working files in your own team folder. Do not edit another team's directory.

## Academic integrity rule

Use outside sources, but cite them. Do not paste unattributed content into team documents.
